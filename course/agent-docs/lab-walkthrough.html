LAB 0 Environment Setup

Module 1 · Smoke test and verification

Smoke Test Agent Creation

What This Lab Does

Verifies your environment is ready: Python 3.11+, the deepagents and langchain-openai packages, and a valid OpenAI API key. Then it creates a minimal agent and confirms it can respond — a quick “hello world” for the course.

Architecture

в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ в”Ӯ check_setup() в”Ӯ Checks: Python, packages, API key в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ в”Ӯ all OK? в–ј в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ в”Ӯ test_agent() в”Ӯ create_deep_agent() вҶ’ invoke() вҶ’ print в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ

Code Walkthrough

The simplest possible agent — no tools, just a system prompt. invoke() sends a message and returns the response. The thread_id identifies the conversation for stateful follow-ups.
from deepagents import create_deep_agent from langchain_core.messages import HumanMessage agent = create_deep_agent( model="openai:gpt-4o-mini", system_prompt="You are a test agent. Respond with exactly: SETUP OK" ) result = agent.invoke( {"messages": [HumanMessage(content="Confirm setup")]}, config={"configurable": {"thread_id": "setup-test"}} )

What You Learn

  • create_deep_agent() is the factory function for all agents
  • Every invocation needs a thread_id for conversation state
  • The simplest agent is just a model + system prompt, no tools

LAB 1 Your First Agent

Module 1 · Custom tools, tool calling, thread identity

@tool Decorator Tool Calling Thread ID

What This Lab Does

Builds an agent with two custom tools — a calculator and a clock. The agent decides when to call each tool based on the user's question. You'll see the LLM reason about whether math or time info is needed, then invoke the right tool automatically.

Architecture

в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ в”Ӯ User в”Ӯв”Җв”Җв”Җв”Җв”ҖвҶ’в”Ӯ Agent (LLM) в”Ӯ в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ в””в”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”ҳ в”Ӯ в”Ӯ decides which tool в”Ӯ в”Ӯ в”Ңв”Җв”Җв”Җв”Җв”Җв–јв”Җв”Җв”җ в”Ңв”Җв”Җв–јв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ в”Ӯ calc() в”Ӯ в”Ӯ get_time() в”Ӯ в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ

Code Walkthrough

The @tool decorator turns a plain Python function into something the LLM can call. The docstring becomes the tool description the model reads to decide when to use it. Args in the docstring define the parameter schema.
@tool def calculator(expression: str) -> str: """Evaluate a mathematical expression. Supports: +, -, *, /, **, sqrt, sin, cos, log, pi, e Args: expression: A math expression, e.g. "2 + 3 * 4" """ allowed_names = {k: v for k, v in math.__dict__.items() if not k.startswith("__")} allowed_names.update({"abs": abs, "round": round}) result = eval(expression, {"__builtins__": {}}, allowed_names) return f"Result: {result}" @tool def get_current_time() -> str: """Get the current date and time.""" return f"Current date/time: {datetime.now():%Y-%m-%d %H:%M:%S (%A)}"
Pass tools as a list to create_deep_agent(). The system prompt tells the model how to use its tools (e.g., "always use calculator for math").
agent = create_deep_agent( model="openai:gpt-4o-mini", tools=[calculator, get_current_time], system_prompt="""You are a helpful assistant with a calculator and clock. ## Rules - For ANY math question, ALWAYS use the calculator tool - Show your work: state what you're calculating and why """ )

What You Learn

  • @tool decorator — how to give agents capabilities
  • create_deep_agent() — the factory function with tools= parameter
  • Tool calling — the LLM decides when and how to use each tool
  • Thread ID — conversation identity for stateful multi-turn chats

LAB 2 Research Agent

Module 2 · Web search, filesystem notes, planning

Web Search Built-in File I/O Planning (Todos) Context Offloading

What This Lab Does

Creates a research agent that follows a structured workflow: plan research steps, search the web, write notes to files, and produce a polished report. Only web_search is passed explicitly — filesystem tools (write_file, read_file) and planning tools (write_todos) are built-in to the framework and available automatically. The key insight is context offloading — the agent saves research to files so it doesn't lose information when the conversation gets long.

Architecture

в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ в”Ӯ Research Agent в”Ӯ в”Ӯ в”Ӯ в”Ӯ Explicit tool: в”Ӯ в”Ӯ web_search вҶ’ search for information в”Ӯ в”Ӯ в”Ӯ в”Ӯ Built-in tools (provided by framework): в”Ӯ в”Ӯ write_todos вҶ’ plan research steps в”Ӯ в”Ӯ write_file вҶ’ save notes to /research/ в”Ӯ в”Ӯ read_file вҶ’ read back notes for synthesis в”Ӯ в”Ӯ в”Ӯ в”Ӯ Workflow: в”Ӯ в”Ӯ plan вҶ’ search вҶ’ write notes вҶ’ synthesize report в”Ӯ в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ Files created: /research/topic_1.md вҶҗ notes per topic /research/sources.md вҶҗ URLs and citations /output/report.md вҶҗ final polished report

Code Walkthrough

Note that tools=[web_search] only lists the custom tool. The framework automatically provides built-in tools like write_file, read_file, and write_todos. The system prompt references these built-in tools in its workflow. The rule "write notes IMMEDIATELY after each search" prevents information loss when the context window fills up.
research_agent = create_deep_agent( model="openai:gpt-4o-mini", tools=[web_search], system_prompt="""You are a research agent that produces well-organized reports. ## WORKFLOW (ALWAYS follow this order) 1. **Plan**: Use write_todos to break the research into 3-5 steps 2. **Research**: Search for information (2-3 searches per topic) 3. **Take Notes**: Write findings to /research/ as markdown files 4. **Synthesize**: Read notes and write a final report to /output/report.md 5. **Deliver**: Present a summary to the user with key findings ## Research Quality Rules - Write notes IMMEDIATELY after each search вҖ” don't hold in memory - Always include source URLs - Distinguish facts from opinions """ )

What You Learn

  • Built-in tools — write_file, read_file, and write_todos come from the framework automatically
  • Only custom tools need tools=[...] — here, just web_search
  • Context offloading — research saved to files, not just held in memory
  • Structured output — organized report at /output/report.md

LAB 3A First Sub-Agent

Module 3 · Orchestrator / worker delegation

Orchestrator / Worker task() Delegation Context Isolation

What This Lab Does

Introduces the orchestrator / sub-agent pattern. The orchestrator has no tools — it must delegate work to a researcher sub-agent that has web search. This forces clean separation: the orchestrator plans and synthesizes, while the sub-agent executes. Each sub-agent gets a fresh, isolated context.

Architecture

в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ в”Ӯ Orchestrator в”Ӯ No tools вҖ” must delegate в”Ӯ (gpt-4o-mini) в”Ӯ в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ в”Ӯ task(subagent_type="researcher") в–ј в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ в”Ӯ Researcher Sub-Agent в”Ӯ Has web_search tool в”Ӯ (gpt-4o-mini) в”Ӯ Fresh isolated context в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ в”Ӯ final answer only в–ј Orchestrator synthesizes and presents to user

Code Walkthrough

Sub-agents are defined as dictionaries with name, description, system_prompt, tools, and model. The orchestrator's tools=[] forces it to use task() for all work.
agent = create_deep_agent( model="openai:gpt-4o-mini", tools=[], # No tools вҖ” MUST delegate subagents=[ { "name": "researcher", "description": ( "Use this agent to search the web and gather " "information on any topic." ), "system_prompt": ( "You are a research agent. Use web_search to find " "information.\nReturn a concise summary.\n" "Search 2-3 times for thorough coverage." ), "tools": [web_search], "model": "openai:gpt-4o-mini", } ], system_prompt="""You are an orchestrator agent. You CANNOT search the web yourself. You MUST delegate ALL research to the "researcher" sub-agent. ## WORKFLOW 1. Analyze the user's question 2. Delegate: task(description="...", subagent_type="researcher") 3. Receive findings 4. Synthesize and present a clear answer""" )

What You Learn

  • Sub-agents get isolated context (no cross-contamination)
  • Only the sub-agent's final message returns (prevents context bloat)
  • The task() tool is how the orchestrator delegates work
  • Toolless orchestrators are forced into clean delegation patterns

LAB 3B Parallel Sub-Agents

Module 3 · Three specialists running simultaneously

Parallel Execution Specialist Agents Synthesis File Sharing

What This Lab Does

The orchestrator launches three specialist sub-agents in parallel — a tech researcher, a market researcher, and a pros/cons analyst. All three run simultaneously (multiple task() calls in a single LLM response), then the orchestrator synthesizes their independent findings into a unified report.

Architecture

в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ в”Ӯ Orchestrator в”Ӯ в””в”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”ҳ в”Ӯ в”Ӯ в”Ӯ вҶҗ 3 task() calls in ONE response в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв–јв”Җв”Җв”җ в”Ңв”Җв”Җв”Җв–јв”Җв”Җв”Җв”Җв”җ в”Ңв”Җв–јв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ в”Ӯ tech в”Ӯ в”Ӯ market в”Ӯ в”Ӯ pros_cons в”Ӯ All run in в”Ӯresearcherв”Ӯ в”Ӯresearcherв”Ӯ в”Ӯ analyst в”Ӯ PARALLEL в””в”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”ҳ в””в”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”ҳ в””в”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”ҳ в”Ӯ в”Ӯ в”Ӯ в–ј в–ј в–ј /research/ /research/ /research/ technical.md market.md pros_cons.md в”Ӯ в”Ӯ в”Ӯ в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”јв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ в–ј Orchestrator reads all вҶ’ synthesizes report

Code Walkthrough

Each specialist has a focused description and system_prompt. The orchestrator's system prompt says "call all three task() tools in a SINGLE response" which triggers parallel execution.
subagents = [ { "name": "tech_researcher", "description": "Researches TECHNICAL aspects вҖ” architecture, " "APIs, code-level details.", "tools": [web_search], "model": "openai:gpt-4o-mini", }, { "name": "market_researcher", "description": "Researches MARKET aspects вҖ” adoption, ecosystem, " "pricing, jobs.", "tools": [web_search], ... }, { "name": "pros_cons_analyst", "description": "Analyzes STRENGTHS and WEAKNESSES вҖ” tradeoffs, " "best/worst use cases.", "tools": [web_search], ... }, ]
The key line: "Call all three task() tools in a SINGLE response for parallel execution!" When the LLM emits multiple tool calls in one response, they run concurrently.
system_prompt="""You are an orchestrator that produces comprehensive reports. ## WORKFLOW 1. Identify the topic and 3 research angles 2. Launch ALL THREE sub-agents IN PARALLEL: - tech_researcher for technical analysis - market_researcher for market analysis - pros_cons_analyst for strengths/weaknesses Call all three task() tools in a SINGLE response! 3. After all results return, read files in /research/ 4. Synthesize into a unified report """

What You Learn

  • Parallel execution — multiple task() calls in one LLM response
  • Specialist agents — each focuses on a single domain for deeper results
  • Synthesis — the orchestrator combines all perspectives into a unified report
  • File sharing — agents write to /research/, orchestrator reads

LAB 3C Filesystem as Communication Bus

Module 3 · Sequential pipeline, file-based handoff

File-Based Communication Sequential Pipeline Calculator Tool

What This Lab Does

Demonstrates the filesystem as a message bus between agents. A data_collector writes structured data to /data/, then an analyst reads those files, performs calculations, and writes results to /analysis/. The orchestrator reads the final analysis and presents it. Each agent only knows about files, not other agents.

Architecture

data_collector analyst orchestrator в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ в”Ӯ Generates в”Ӯ в”Ӯ Reads /data/ в”Ӯ в”Ӯ Reads в”Ӯ в”Ӯ data в”Ӯв”Җв”Җв”Җв”Җв”Җв”ҖвҶ’в”Ӯ Calculates в”Ӯв”Җв”Җв”Җв”Җв”Җв”ҖвҶ’в”Ӯ/analysis/в”Ӯ в”Ӯ tables в”Ӯ files в”Ӯ Writes в”Ӯ files в”Ӯ Presents в”Ӯ в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ в”Ӯ /analysis/ в”Ӯ в”Ӯ to user в”Ӯ в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ /data/cloud_pricing.md вҶ’ /analysis/results.md вҶ’ user

Code Walkthrough

The data_collector creates structured markdown tables. The analyst reads those files and has a calculate tool for math. The orchestrator's system prompt defines the sequential workflow: "FIRST data_collector, THEN analyst, FINALLY you."
subagents = [ { "name": "data_collector", "description": "Collects and structures raw data. " "Writes to /data/ as markdown tables.", "system_prompt": """Write data as MARKDOWN TABLES to /data/[topic].md Always include headers and units. Other agents will read your output!""", "tools": [], }, { "name": "analyst", "description": "Reads data from /data/, performs calculations, " "writes conclusions to /analysis/.", "system_prompt": """Read data files from /data/. Calculate with the calculate tool. Write analysis to /analysis/results.md ALWAYS show your calculation steps!""", "tools": [calculate], }, ] # Orchestrator's workflow (sequential): # 1. FIRST: data_collector вҶ’ /data/ # 2. THEN: analyst вҶ’ /analysis/ # 3. FINALLY: read /analysis/ вҶ’ present to user

What You Learn

  • The filesystem is the message bus — agents communicate through files
  • Pipeline pattern: produce → consume → produce → consume
  • Each agent only knows about files, not about other agents directly
  • Data persists in files even after conversation summarization

LAB 4 Financial Deep Research

Module 4 · Capstone — 4-agent multi-phase orchestration

Parallel + Sequential 4 Specialist Agents File Communication Multi-Phase Financial Tools

What This Lab Does

A production-grade financial analysis system with 4 specialist agents running in a phased workflow. Phase 1 launches market_data_agent and news_analyst in parallel. Phase 2 sends results to finance_analyst for quantitative analysis. Phase 3 has report_writer synthesize everything into a professional investment report.

Architecture

в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ в”Ӯ ORCHESTRATOR в”Ӯ в””в”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”ҳ в”Ӯ в”Ӯ в•җв•җв•җ Phase 1: Data Collection (PARALLEL) в•җв•җв•җ в”Ӯ в”Ӯ в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–јв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–јв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ в”Ӯ market_data в”Ӯ в”Ӯ news_analyst в”Ӯ в”Ӯ get_stock_dataв”Ӯ в”Ӯ search_financial в”Ӯ в”Ӯ вҶ’ /data/marketв”Ӯ в”Ӯ вҶ’ /data/news/ в”Ӯ в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ в”Ӯ в”Ӯ в•җв•җв•җ Phase 2: Analysis (SEQUENTIAL) в•җв•җв•җв•җв•җв•җв•җв•җв•җ в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–јв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ в”Ӯ finance_analyst в”Ӯ в”Ӯ calc metrics в”Ӯ в”Ӯ вҶ’ /data/analysis/в”Ӯ в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ в”Ӯ в•җв•җв•җ Phase 3: Report (SEQUENTIAL) в•җв•җв•җв•җв•җв•җв•җв•җв•җв•җв•җ в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–јв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ в”Ӯ report_writer в”Ӯ в”Ӯ reads all /data/в”Ӯ в”Ӯ вҶ’ /report/ в”Ӯ в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ

Code Walkthrough
