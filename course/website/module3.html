3.1 — Your First Sub-Agent

The task() Tool

In a multi-agent system, the orchestrator does not do work directly. Instead, it delegates by calling a built-in task() tool:

# The orchestrator calls this internally when it decides to delegate
task(
    description="Research the latest trends in quantum computing",
    subagent_type="researcher"
)

The framework handles the rest: it spins up the named sub-agent in an isolated context window, passes the description as the task, and returns the sub-agent's final answer to the orchestrator.

Key Insight
The orchestrator never sees the sub-agent's intermediate tool calls or reasoning. It only receives the final result. This is what keeps the orchestrator's context clean.

# lab3a_first_subagent.py
"""
LAB 3a: Your First Sub-Agent
An orchestrator that delegates research to a specialist sub-agent.
The orchestrator has NO tools — it can only delegate.
"""
import httpx
from langchain_core.tools import tool
from langchain_core.messages import HumanMessage
from deepagents import create_deep_agent

@tool
def web_search(query: str) -> str:
    """Search the web for information.

    Args:
        query: The search query string
    """
    try:
        response = httpx.get(
            "https://api.duckduckgo.com/",
            params={"q": query, "format": "json", "no_html": "1"},
            timeout=10
        )
        data = response.json()
        results = []
        if data.get("Abstract"):
            results.append(f"Summary: {data['Abstract']}")
        for topic in data.get("RelatedTopics", [])[:5]:
            if isinstance(topic, dict) and "Text" in topic:
                results.append(topic["Text"])
        return "\n".join(results) if results else "No results found."
    except Exception as e:
        return f"Search error: {e}"

Step 2: Define the Sub-Agent

# Define the researcher sub-agent specification
researcher_spec = {
    "name": "researcher",
    "description": "A research specialist that searches the web and summarizes findings",
    "system_prompt": """You are a research specialist agent.

## Your Role
- Search the web for information on the given topic
- Synthesize findings into a clear, structured summary
- Always cite what you found from search results

## Rules
- Use web_search for EVERY factual claim
- Maximum 3 searches per task
- Return a concise summary (under 300 words)
- Structure output with bullet points""",
    "tools": [web_search],
    "model": "openai:gpt-4o-mini"
}

Step 3: Create the Orchestrator (No Tools!)

# The orchestrator has NO tools — it can ONLY delegate
orchestrator = create_deep_agent(
    model="openai:gpt-4o",
    tools=[],  # No tools! Forces delegation
    subagents=[researcher_spec],
    system_prompt="""You are an orchestration agent.

## Your Role
- You answer user questions by delegating to your sub-agents
- You have NO tools of your own — you MUST delegate
- You have one sub-agent: "researcher" — use it for any factual question

## Workflow
1. Receive user question
2. Delegate to researcher sub-agent via task()
3. Review the researcher's findings
4. Deliver a polished final answer to the user"""
)

Step 4: Run It

def ask(question: str):
    """Send a question to the orchestrator."""
    print(f"\n{'='*60}")
    print(f"USER: {question}")
    print("=" * 60)

    result = orchestrator.invoke(
        {"messages": [HumanMessage(content=question)]},
        config={"configurable": {"thread_id": "lab3a"}}
    )

    for msg in reversed(result["messages"]):
        if msg.type == "ai" and msg.content:
            print(f"\nORCHESTRATOR: {msg.content}")
            return

What Happens Under the Hood

Step 1: User sends question to ORCHESTRATOR ↓ Step 2: Orchestrator has no tools, decides to delegate ↓ Step 3: Orchestrator calls task(description="Research quantum...", subagent_type="researcher") ↓ Step 4: Framework creates RESEARCHER sub-agent with its own context ↓ Step 5: Researcher calls web_search("quantum computing applications") Researcher calls web_search("quantum computing use cases 2025") Researcher synthesizes findings into summary ↓ Step 6: Researcher's final summary is returned to ORCHESTRATOR (all intermediate searches stay isolated) ↓ Step 7: Orchestrator formats and delivers polished answer to USER

3.2 — Parallel Sub-Agents

Why Parallel?

When a complex question has independent aspects, you can research them all at the same time. The orchestrator launches multiple task() calls in a single response, and the framework runs them in parallel.

For example, the question "Should our startup use Rust or Go?" has three independent aspects:

  1. Technical comparison — language features, performance, ecosystem
  2. Market analysis — adoption trends, job market, community
  3. Pros and cons — trade-offs for a startup context

Three specialist agents can each tackle one aspect simultaneously.

Why Parallel Matters

Without parallel execution, 3 sub-agents running sequentially would take 3x as long. With parallel execution, they all run simultaneously and the total time equals the slowest agent, not the sum.

3.3 — Filesystem as Communication Bus

The Problem: How Do Agents Share Data?

In sections 3.1 and 3.2, the orchestrator received sub-agent results directly through task() return values. But what about more complex pipelines where:

  • Agent A produces large datasets that would overwhelm Agent B's context
  • Agents run in different phases and need to pass work products
  • You want an audit trail of intermediate results

The answer: use the filesystem as a communication bus. Agent A writes files. Agent B reads them.

/workspace/ ├── data/ ← data_collector writes here │ ├── raw_metrics.json │ └── market_data.json └── analysis/ ← analyst reads /data/, writes here └── final_analysis.md

LAB 3c: Filesystem as Communication Bus
Agent A (data_collector) writes data files.
Agent B (analyst) reads those files and produces analysis.

@tool
def calculator(expression: str) -> str:
    """Evaluate a mathematical expression.

    Args:
        expression: A math expression, e.g. "2 + 3 * 4" or "sqrt(144)"
    """
    allowed_names = {k: v for k, v in math.__dict__.items() if not k.startswith("__")}
    allowed_names.update({"abs": abs, "round": round})
    try:
        result = eval(expression, {"__builtins__": {}}, allowed_names)
        return f"Result: {result}"
    except Exception as e:
        return f"Error: {e}"

@tool
def write_analysis_file(filename: str, content: str) -> str:

    Args:
        filename: Name of the file to create (e.g., "report.md")

Data Flow: File-Based Communication

Phase 1: DATA COLLECTION ┌──────────────┐ ┌──────────────────────────────┐ │ data_collector│ ──→ │ /workspace/data/ │ │ │ │ frameworks.json │ │ web_search() │ │ performance_data.json │ │ write_data() │ │ community_stats.json │ └──────────────┘ └──────────────┬─────────────────┘ │ │ Files on disk │ (persist between agents) │ Phase 2: ANALYSIS ▼ ┌──────────────┐ ┌──────────────────────────────┐ │ analyst │ ←── │ /workspace/data/ │ │ │ │ (reads all files) │ │ read_data() │ └──────────────────────────────┘ │ calculator() │ │ │ ──→ ┌──────────────────────────────┐ │ write_ │ │ /workspace/analysis/ │ │ analysis() │ │ report.md │ └──────────────┘ └──────────────────────────────┘
Why Files Instead of Return Values?
  • Scalability — Return values go into the orchestrator's context window. Large datasets would overflow it. Files don't.
  • Audit trail — Every intermediate artifact is saved to disk. You can inspect exactly what each agent produced.
  • Decoupling — Agent A doesn't need to know about Agent B. It just writes to a directory. Any number of downstream agents can read from it.

def ask(question: str, thread_id: str = "lab3"):
    """Send a question through the full multi-agent pipeline."""
    print(f"\n{'='*60}")
    print(f"QUESTION: {question}")
    print("=" * 60)

    result = orchestrator.invoke(
        {"messages": [HumanMessage(content=question)]},
        config={"configurable": {"thread_id": thread_id}}
    )

    # Print the orchestrator's final answer
    for msg in reversed(result["messages"]):
        if msg.type == "ai" and msg.content:
            print(f"\nFINAL ANSWER:\n{msg.content}")
            break

    # Show pipeline artifacts
    print(f"\n{'─'*40}")
    print("Pipeline artifacts:")
    for d, label in [(RESEARCH_DIR, "research"), (OUTPUT_DIR, "output")]:
        for f in os.listdir(d):
            fp = os.path.join(d, f)
            print(f"  [{label}] {fp} ({os.path.getsize(fp)} bytes)")

Key Patterns Learned

PatternHow It WorksUse When
Single delegation task(desc, subagent_type="x") Simple subtask that one specialist can handle
Parallel delegation Multiple task() calls in one response Independent aspects that can be researched simultaneously
File-based communication Agent A writes → Agent B reads Data pipelines, large datasets, audit trails
Specialist agents Different tools per sub-agent Separation of concerns, focused expertise
Orchestrator has no tools Forces delegation to sub-agents Clean architecture, prevents orchestrator from doing work directly
Architecture Rule of Thumb
  • Orchestrator = no tools, only delegates. Keeps context clean.
  • Sub-agents = specialized tools, focused system prompts. Each does one thing well.
  • Filesystem = the communication bus. Write to share, read to consume.
