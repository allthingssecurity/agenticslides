2.1 — Tools & Backends

Backend Architecture

Every Deep Agent needs a backend — the storage layer that holds files, state, and execution context. The architecture is pluggable via a protocol interface:

┌─────────────────────────────────────────────────────┐ │ AGENT │ │ │ │ LLM ←→ Tools ←→ BackendProtocol │ │ │ │ │ ┌────────────┼────────────┐ │ │ ↓ ↓ ↓ │ │ ┌─────────────┐ ┌─────────┐ ┌──────────────┐ │ │ │ State │ │Filesystem│ │ LocalShell │ │ │ │ Backend │ │ Backend │ │ Backend │ │ │ │ (in-memory) │ │ (disk) │ │ (disk+shell) │ │ │ └─────────────┘ └─────────┘ └──────────────┘ │ └─────────────────────────────────────────────────────┘

Available Backends

BackendStorageUse Case
StateBackendIn-memory dictTesting, simple agents, no persistence needed
FilesystemBackendLocal diskFile I/O without shell access (safer)
LocalShellBackendDisk + shellFull development environment (default)
StoreBackendLangGraph storeRemote/distributed deployments
CompositeBackendMultiple backendsCombine backends for different capabilities

Filesystem Tools

When an agent uses a filesystem-capable backend, the FilesystemMiddleware automatically injects these tools:

ToolDescriptionExample
lsList directory contentsls(path="/research")
read_fileRead a file's contentsread_file(path="/notes.md")
write_fileCreate or overwrite a filewrite_file(path="/output/report.md", content="...")
edit_fileSurgical edit with old/new stringsedit_file(path="/config.json", old="v1", new="v2")
globFind files matching a patternglob(pattern="**/*.md")
grepSearch file contentsgrep(pattern="TODO", path="/src")
executeRun a shell command (LocalShell only)execute(command="pip install pandas")

Backend Configuration

from deepagents import create_deep_agent
from deepagents.backends import (
    StateBackend,
    FilesystemBackend,
    LocalShellBackend
)

# Option 1: In-memory only (no filesystem tools)
agent_simple = create_deep_agent(
    model="openai:gpt-4o",
    backend=StateBackend()
)

# Option 2: Filesystem access (read/write files, no shell)
agent_fs = create_deep_agent(
    model="openai:gpt-4o",
    backend=FilesystemBackend(root="./workspace")
)

# Option 3: Full local environment (default)
agent_full = create_deep_agent(
    model="openai:gpt-4o",
    backend=LocalShellBackend(root="./workspace")
)

2.2 — Planning with Todos

Why Planning Matters

Without planning, an agent takes a random walk through solution space. With planning, it follows a structured path:

WITHOUT PLANNING WITH PLANNING Start Start ↓ ↓ Search X Plan: [1] [2] [3] [4] [5] ↓ ↓ Oops, wrong direction Step 1: Research topic A ✓ ↓ ↓ Try something else Step 2: Research topic B ✓ ↓ ↓ Search Y Step 3: Analyze findings ✓ ↓ ↓ Forget original goal Step 4: Write report ✓ ↓ ↓ Incomplete result Step 5: Review & deliver ✓ ↓ Complete, high-quality result

The write_todos Tool

Every Deep Agent has a built-in write_todos tool. When invoked, the agent creates a structured task list:

# The agent calls write_todos with a JSON task list
write_todos({
    "todos": [
        {"id": 1, "task": "Search for Python backend frameworks", "status": "in_progress"},
        {"id": 2, "task": "Search for JavaScript backend frameworks", "status": "pending"},
        {"id": 3, "task": "Compare performance benchmarks", "status": "pending"},
        {"id": 4, "task": "Write comparison report", "status": "pending"},
        {"id": 5, "task": "Summarize key findings", "status": "pending"}
    ]
})

The agent updates the status as it completes each step: "pending" → "in_progress" → "done".

Planning Demo

from deepagents import create_deep_agent
from langchain_core.messages import HumanMessage

planner_agent = create_deep_agent(
    model="openai:gpt-4o",
    system_prompt="""You are a research analyst. You ALWAYS plan before acting.

## Rules
1. FIRST use write_todos to create a 3-5 step plan
2. Work through each step systematically
3. Update todo status as you complete each step
4. Write findings to files as you go
5. Deliver a summary when all steps are done"""
)

result = planner_agent.invoke(
    {"messages": [HumanMessage(content="Research the pros and cons of microservices vs monoliths")]},
    config={"configurable": {"thread_id": "planning-demo"}}
)

Internal Agent Behavior

Here's what happens inside the agent when it plans:

User: "Research microservices vs monoliths" ↓ LLM thinks: "I need to plan first" ↓ Tool call: write_todos([ {id: 1, task: "Research microservices pros/cons", status: "in_progress"}, {id: 2, task: "Research monolith pros/cons", status: "pending"}, {id: 3, task: "Compare and contrast", status: "pending"}, {id: 4, task: "Write report to file", status: "pending"} ]) ↓ LLM: "Step 1 — Researching microservices..." ↓ Tool call: write_file("/research/microservices.md", "...") ↓ Tool call: write_todos([...update step 1 → done, step 2 → in_progress...]) ↓ LLM: "Step 2 — Researching monoliths..." ↓ ... continues until all steps are done ...
Why Structured Planning Works
The todo list acts as a persistent anchor in the conversation. Even if early messages get summarized, the current todo state keeps the agent on track. Each write_todos call gives the LLM a complete picture of progress.

2.3 — Context Management

The Context Window Problem

Every LLM has a finite context window. As the agent works, the conversation fills up:

Context Window Usage Over Time: 0% ├────────────────────────────────────────┤ Empty 25% ├██████████──────────────────────────────┤ After a few tool calls 50% ├████████████████████──────────────────────┤ After research phase 75% ├██████████████████████████████──────────┤ Writing report 100% ├████████████████████████████████████████┤ FULL — LLM errors! Problem: Once full, the LLM can't process new information. Older messages get truncated and WORK IS LOST.

SummarizationMiddleware

The SummarizationMiddleware automatically compresses the conversation when it gets too long:

from deepagents import create_deep_agent
from deepagents.middleware import SummarizationMiddleware

agent = create_deep_agent(
    model="openai:gpt-4o",
    middleware=[
        SummarizationMiddleware(
            model="openai:gpt-4o-mini",    # Cheap model for summarization
            threshold=0.7,                 # Trigger at 70% context usage
            preserve_recent=5,             # Keep last 5 messages intact
        )
    ]
)

Before and After Summarization

BEFORE SUMMARIZATION AFTER SUMMARIZATION ┌──────────────────────────┐ ┌──────────────────────────┐ │ System Prompt │ │ System Prompt │ │ User: "Research AI..." │ │ │ │ AI: "I'll plan first..." │ │ [Summary]: The agent │ │ Tool: write_todos(...) │ ──────→│ researched AI frameworks, │ │ Tool: search("langchain") │ │ took notes on 5 tools, │ │ Tool result: {...} │ │ and is now writing the │ │ AI: "Found LangChain..." │ │ final report. │ │ Tool: write_file(...) │ │ │ │ Tool: search("crewai") │ │ AI: "Now writing report" │ │ Tool result: {...} │ │ Tool: read_file(...) │ │ AI: "Found CrewAI..." │ │ Tool: write_file(...) │ │ Tool: write_file(...) │ │ AI: "Report complete!" │ │ ...20 more messages... │ └──────────────────────────┘ │ AI: "Now writing report" │ │ Tool: read_file(...) │ 70% context → 25% context │ Tool: write_file(...) │ Agent can keep working! │ AI: "Report complete!" │ └──────────────────────────┘

Key Insight: Files Survive Summarization

This is the critical pattern:

The Write-Then-Summarize Pattern
  1. Agent writes research to files as it goes
  2. Summarization compresses the conversation (what was discussed)
  3. But the files on disk are untouched — no data is lost
  4. Agent can read files back when it needs the data again

This is why filesystem-based agents can handle arbitrarily complex tasks — they use files as external memory.

The Write-Then-Summarize System Prompt Pattern

system_prompt = """You are a research agent with context management.

## CRITICAL RULES FOR CONTEXT MANAGEMENT
1. ALWAYS write findings to files IMMEDIATELY after each search
2. Never keep large amounts of data only in conversation
3. Use /research/ for raw notes, /output/ for final deliverables
4. When you need old data, READ it from files — don't rely on memory

## Why This Matters
Your conversation history may be summarized to save space.
When that happens, anything only in chat messages may be lost.
But files on disk ALWAYS survive. Write early, write often.
"""
Mental Model
Think of the context window as RAM and the filesystem as disk. Just like a real computer, you save important data to disk so it survives when RAM is cleared.

2.4 — Lab 2: Research Agent

Lab 2: Research Agent

Goal: Build a research agent that searches the web, takes structured notes, and produces a polished report — all using the patterns from this module.

Time: 15 minutes

Download lab2_research_agent.py

Step 1: Web Search Tool

First, we create a tool that searches the web using the DuckDuckGo API:

# lab2_research_agent.py
import json
import httpx
from langchain_core.tools import tool
from langchain_core.messages import HumanMessage
from deepagents import create_deep_agent

@tool
def web_search(query: str, num_results: int = 5) -> str:
    """Search the web for information.

    Args:
        query: The search query
        num_results: Number of results to return (default 5)
    """
    try:
        response = httpx.get(
            "https://api.duckduckgo.com/",
            params={"q": query, "format": "json", "no_html": 1},
            timeout=10
        )
        data = response.json()
        results = []
        if data.get("Abstract"):
            results.append({
                "title": data.get("Heading", "Main Result"),
                "snippet": data["Abstract"],
                "source": data.get("AbstractURL", "")
            })
        for topic in data.get("RelatedTopics", [])[:num_results]:
            if isinstance(topic, dict) and "Text" in topic:
                results.append({
                    "title": topic.get("Text", "")[:80],
                    "snippet": topic.get("Text", ""),
                    "source": topic.get("FirstURL", "")
                })
        if not results:
            return f"No results found for: {query}"
        return json.dumps(results, indent=2)
    except Exception as e:
        return f"Search error: {e}"

Step 2: Research Agent with System Prompt

Now we create the agent with a detailed system prompt that enforces the research workflow:

research_agent = create_deep_agent(
    model="openai:gpt-4o",
    tools=[web_search],
    system_prompt="""You are a research agent that produces well-organized reports.

## WORKFLOW (ALWAYS follow this order)
1. **Plan**: Use write_todos to break the research into 3-5 steps
2. **Research**: Search for information on each aspect (2-3 searches per topic)
3. **Take Notes**: Write findings to /research/ as structured markdown files
4. **Synthesize**: Read your notes and write a final report to /output/report.md
5. **Deliver**: Present a summary to the user with key findings

## File Organization
/research/
├── topic_1.md
├── topic_2.md
└── sources.md
/output/
└── report.md

## Research Quality Rules
- Search at least 2-3 times per topic from different angles
- Write notes IMMEDIATELY after each search
- Always include source URLs
- Distinguish facts from opinions
- If information conflicts, note the discrepancy
"""
)

Step 3: Run the Research

def research(question: str, thread: str = "lab2"):
    print(f"RESEARCH QUERY: {question}")
    result = research_agent.invoke(
        {"messages": [HumanMessage(content=question)]},
        config={"configurable": {"thread_id": thread}}
    )
    for msg in reversed(result["messages"]):
        if msg.type == "ai" and msg.content:
            print(msg.content)
            return

if __name__ == "__main__":
    research(
        "Compare Python vs JavaScript for backend development in 2025. "
        "Cover: performance, ecosystem, developer experience, and job market."
    )
